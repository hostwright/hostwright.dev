A project is defined as much by what it refuses to do as by what it builds.
Hostwright's non-goals are deliberate, and each has a reason. The authoritative,
exhaustive list lives in [Limitations](/docs/reference/limitations/); this page
explains the reasoning.

![Copy, adapt, reject framing for which container-tooling ideas Hostwright keeps, reshapes, or refuses.](/docs/diagrams/12-copy-adapt-reject.svg)

## Copy, adapt, reject

Hostwright studies prior art and decides explicitly what to take:

| Source idea | Decision | What Hostwright does |
| --- | --- | --- |
| Kubernetes controllers | **Adapt** | A local observe–diff–plan–apply–observe loop. |
| Kubernetes API server | **Reject (for now)** | Single-host control needs no distributed API server. |
| Container Runtime Interface (CRI) | **Study, don't implement** | Inspiration for the adapter boundary, not a shim. |
| Docker Compose | **Adapt lightly** | A readable local stack format, not full parity. |
| Docker API compatibility | **Reject as core** | Already covered elsewhere; distracts from controller-first design. |
| Apple container CLI | **Adopt initially** | Structured output behind the adapter while native APIs settle. |
| Containerization APIs | **Adopt when stable** | A Swift-native runtime path, later. |

## Why CRI is research-only

CRI is a strong design reference for a runtime boundary, which is exactly how
Hostwright uses it — as inspiration for the [RuntimeAdapter](/docs/concepts/runtime-adapter/).
Implementing CRI itself would mean committing to Kubernetes' contract and node-agent
semantics that single-host Hostwright does not need. External orchestration
compatibility (CRI shims, Kubernetes node behavior, Docker API shims,
Testcontainers targets) is research-only and rejected for the current core.

## Why full Compose parity is rejected

Compose is large. Chasing 100% parity would turn Hostwright into a compatibility
project instead of a control plane. Hostwright takes the readable-local-stack idea
and stops there, deliberately narrower than Compose. The
[stack importer](/docs/tasks/stack-import/) converts a narrow reviewed subset and
fails closed on everything else.

## Why a cloud control plane is out

Hostwright is single-host by design. A hosted control plane, multi-Mac
orchestration, and remote state are a different product with different failure
modes. Multi-host platform work is research-only: remote host agents, membership,
peer discovery, state replication, remote mutation, and scheduler APIs are all
rejected for the current core.

## Why GPU and accelerator claims are research-only

Apple silicon acceleration — GPU/ANE scheduling, Metal, Core ML, MLX, PyTorch MPS
in containers — is genuinely interesting and genuinely hard. It is treated as
research, not as a feature. Hostwright will not claim accelerator capabilities it
has not built and tested.

<Aside type="caution" title="Research is not a promise">
Networking, tunnels, DNS, cloud exposure, multi-host, and accelerator support all
have research behind them in the project's design materials. Research informs the
[roadmap](/docs/roadmap/); it does not become a claim until there is working, tested
code. See [Research areas](/docs/roadmap/research/).
</Aside>

## Why destructive cleanup must be conservative

A tool that deletes the wrong thing once loses trust permanently. So cleanup is
ownership-aware, dry-run first, token-confirmed, and never touches resources
Hostwright cannot prove it owns. This is covered in the
[safety model](/docs/concepts/safety-model/).
