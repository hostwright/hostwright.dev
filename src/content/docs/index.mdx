Hostwright is a **Mac-native desired-state control plane** for [Apple container](/docs/concepts/apple-container-boundary/) workloads. Today it can initialize and validate a `hostwright.yaml` manifest, render a non-mutating manifest-level plan, show manifest-level status, and run safe doctor checks. Runtime observation and apply are planned behind one runtime boundary.

It is built for one machine: an Apple silicon Mac running a recent macOS with Apple container installed. It is not a cluster orchestrator, and it does not try to be.

<Callout type="warning" title="Maturity">
This is an early infrastructure project. The design is written down and locked; the
implementation is just beginning. Treat everything here as a description of intent,
not a shipped, production-ready tool. See [Limitations](/docs/reference/limitations/)
and [Roadmap](/#roadmap).
</Callout>

## What you get

- A declarative manifest, [`hostwright.yaml`](/docs/reference/hostwright-yaml/), for local stacks.
- A current non-mutating planning step that does not inspect Apple container.
- A planned [reconciler](/docs/concepts/reconciliation/) that will converge actual state to declared state and report drift after runtime observation exists.
- A single [runtime adapter](/docs/concepts/runtime-adapter/) boundary that all future Apple container behavior must cross.
- A planned [safety model](/docs/concepts/safety-model/) where destruction is explicit and cleanup is ownership-aware.
